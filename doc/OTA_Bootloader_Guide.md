# XBOX USB CDC OTA Bootloader 使用与架构说明文档

## 1. 工程整体定位与框架介绍

本工程（XBOX）是一个基于沁恒 WCH CH32V203 (RISC-V 架构) 的 **USB OTA Bootloader**。
它的核心作用是：**上电后优先接管 MCU，模拟出一个 USB 虚拟串口 (USB CDC) 供电脑识别。如果电脑在指定时间内（目前设定为 3 秒）发送了特定的 OTA 升级指令，它会接收电脑下发的固件数据并将其烧写到单片机的 Flash App 区域中；如果超时未收到指令，它会主动把系统控制权交给 App 区域的真实用户代码。**

### 1.1 存储器 Flash 空间划分
CH32V203 拥有 64K Flash，本 Bootloader 系统将其划分为两部分：
* **Bootloader 区域 (本工程 `XBOX`)**: `24KB`，起始地址 `0x08000000` (逻辑映射为 `0x00000000`)。
  * 负责 USB 通信、Flash 擦写、固件接收以及超时跳转。
* **App 应用程序区域 (你的真实业务代码)**: `40KB`，起始地址 `0x08006000` (逻辑映射为 `0x00006000`)。
  * 负责手柄摇杆逻辑、按键读取等真实业务。未来你所有的新业务功能开发，都应该写在这个区域。

### 1.2 Bootloader 内部逻辑是怎么跑的？
系统的核心数据流向和控制逻辑如下：

1. **系统上电启动 (`User/Main.c`)**:
   * 初始化系统时钟 (`SystemCoreClockUpdate`)。
   * 初始化 `UART1` (TX=PA9, RX=PA10) 用于调试信息输出 (波特率 115200)。
   * 初始化 `TIM2` 定时器，用于提供 1ms 精度的 `OTA_tick` 时基。
   * 初始化硬件 USB 并枚举为 CDC 串口设备。
2. **主循环监控 (`while(1)`)**:
   * 执行 `UART1_DataRx_Deal()`: 尝试将 UART 的数据通过 USB 回传（原本用于透传，现保留用于上传简单的应答帧 ACK/NACK 给上位机）。
   * 执行 `UART1_DataTx_Deal()`: **核心拦截点**。当 USB 接收到电脑下发的数据存入环形缓冲区时，这里会中止原先发往 UART 发送端 (DMA) 的逻辑，而是将数据推入 `OTA_ProcessPacket()` 函数中解析。
   * 执行 `OTA_CheckTimeoutAndJump()`: 如果连续 3000ms 没有检测到有效 OTA 协议握手，清理内存并跳转执行 `Jump_To_App()`（跳往 `0x08006000` 执行业务代码）。
3. **OTA 协议解析 (`User/UART/OTA.c`)**:
   * 判断帧头 `0xAA 0x55`。
   * 依据指令 (`0x01` 擦除页，`0x02` 写入固件)，解包数据并调用底层 `FLASH_ErasePage` / `FLASH_ProgramHalfWord` 操作 `0x08006000` 以后的 Flash 空间。

## 2. 上位机 (PC端) 如何与 Bootloader 交互

电脑端（可以使用 C#/Python/C++ 编写的上位机工具，也可以使用某些支持 Hex 发送的串口助手）连接该虚拟串口后，需按照以下自定义帧格式进行固件下发：

### 2.1 OTA 帧基础格式
所有发送往 MCU USB 串口的数据必须严格遵循此格式（总长度固定或灵活，但受限于 CDC 单包最大 64 字节，故负载区不宜超过 56 字节）：

| 帧头1 (1 Byte) | 帧头2 (1 Byte) | 指令 Cmd (1 Byte) | 负载长度 Len (1 Byte) | 目标地址 Address (4 Bytes，小端) | 收发负载数据 Payload (Len 字节) |
|---|---|---|---|---|---|
| `0xAA` | `0x55` | 见指令表 | 随 Payload 变化 | `0x0800xxxx` 偏移寻址 | 固件 Hex 二进制内容 |

*(注：实际代码中简化去掉了 CRC，如需使用可自行在 `OTA_ProcessPacket` 扩展尾部判定)*

### 2.2 具体指令表 (Cmd) 与范例
* **擦除指令 (`OTA_CMD_ERASE = 0x01`)**
  * **请求 (PC 发送):** `AA 55 01 00 [4字节要擦除的起止地址]` 
    * *例:* 擦除 App 第一页: `AA 55 01 00 00 60 00 08` (即 `0x08006000`)
  * **应答 (MCU 返回):** `AA 55 01 00` (成功 ACK) 或 `AA 55 01 [错误码]` (失败 NACK)
* **烧写固件指令 (`OTA_CMD_WRITE = 0x02`)**
  * **请求 (PC 发送):** `AA 55 02 [Len] [4字节写入地址] [Len 字节的 bin 文件片段内容]`
    * *例:* 在首地址写入 16 字节的数据：`AA 55 02 10 00 60 00 08 [16字节真实内容]`
  * **应答 (MCU 返回):** `AA 55 02 00` (写入成功)
* **强制跳转 App 指令 (`OTA_CMD_JUMP = 0x04`)**
  * **请求 (PC 发送):** `AA 55 04 00`
  * **描述:** 发送此指令后，Bootloader 会立刻停止等待，重启相关外设并执行指针跳转。

## 3. 手册：如何为这个系统开发你的 App 代码？

一旦你将这个 `XBOX` (Bootloader) 烧写进入了 `0x00000000`，**以后开发新的真实业务代码 (App 工程) 时，必须做以下两点核心修改，否则你的 App 跑不起来：**

**(假设你通过 MounRiver 新建了一个叫 `MyGamepad` 的新硬件工程)**

### 第 1 步：修改新工程的链接脚本 (`Ld/Link.ld`)
在你的 App 工程目录中找到 `Link.ld`，把 `FLASH` 的起始地址往后推 24K (即 `0x00006000`)，并将最大可用长度改为剩余的 40K：
```ld
	/* App 的 Link.ld 修改如下 */
	FLASH (rx) : ORIGIN = 0x00006000, LENGTH = 40K
	RAM (xrw) : ORIGIN = 0x20000000, LENGTH = 20K
```
*(这段修改会让编译器自动把你写的业务代码偏移到 0x08006000 这个物理区域，避免与 Bootloader 地址段发生踩踏冲撞。)*

### 第 2 步：重定向中断向量表 (在 `User/Main.c` 中)
因为 Bootloader 占据了系统最开始的 `0x08000000` 绝对原点，但各种硬件中断 (定时器、GPIO等) 触发时，CPU 仍然会去原点找中断执行表。
所以在 RISC-V 架构下，**你必须在 App 工程的 `main()` 函数第一行执行前告诉 CPU 向量表换地方了**。

```c
/* 在你 App 工程的 Main.c 中 */

int main(void)
{
    // ===【必须添加】中断向量表重定位到 0x08006000 ===
    __asm volatile("csrw mtvec, %0" :: "r"(0x00006000));
    
    // 下面写你正常的系统时钟初始化和其他业务
    NVIC_PriorityGroupConfig(NVIC_PriorityGroup_1);
    SystemCoreClockUpdate();
    Delay_Init();
    // ...
}
```

### 第 3 步：如何将写好的 App 刷进板子？
由于 Bootloader 的存在，现在你**不需要再插仿真器/脱机下载器**去刷 App 了！
1. 在 MounRiver 编译你的新 App 工程，生成 `.bin` 或 `.hex` 文件。
2. 将板子通过 USB 连上电脑，识别出虚拟串口。
3. 复位板子，**3 秒内**用你编写好的上位机工具，按照上面第 2 节列出的 OTA 协议，通过这个虚拟串口将 `.bin` 下发过去。
4. 下发完毕后，Bootloader 自动跳转，你的新手柄逻辑即刻生效！
